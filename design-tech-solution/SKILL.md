# 技术方案设计

产出与现有代码库模式一致、满足已确认需求的详细技术设计。

## 工作流程

1. **研究现有代码库**
   - 识别相关模块、服务和数据模型
   - 注意现有模式（如依赖注入、ORM、API 风格）
   - 参考类似功能以保持风格一致

2. **设计整体流程**
   - 将用户故事映射为系统交互
   - 定义入口点（API 端点、事件、CLI 命令）
   - 区分同步与异步处理

3. **设计数据模型**
   - 新实体、字段、关系
   - Schema 变更的迁移策略
   - 数据生命周期（创建、更新、删除、归档）

4. **识别外部依赖**
   - 第三方 API、库、服务
   - 依赖故障时的降级策略
   - 配置和凭据管理

5. **设计功能模块**
   - 拆分为高内聚的模块
   - 定义模块间接口
   - 分配职责（单一职责原则）

6. **明确改动内容**
   - 需要修改 vs 需要新建的文件
   - 每个模块的预期改动复杂度
   - 向后兼容性考虑

7. **设计非功能性需求**
   - 性能：吞吐量、延迟、缓存策略
   - 安全：鉴权/授权、输入校验、密钥处理
   - 可观测性：日志、指标、链路追踪
   - 可靠性：重试、熔断、优雅降级

[实体图或 Schema 定义]

|------|------|----------|----------|
| ... | ... | ... | ... |

## 模块设计

### 模块：[名称]
- 职责：...
- 接口：...
- 内部函数：...

## 改动清单
| 文件 | 类型 | 复杂度 | 说明 |

## 质量检查清单

- [ ] 输入中的每个功能需求都已处理
- [ ] 数据模型能覆盖所有已识别的边界场景
- [ ] 每个外部依赖至少定义了一个降级方案
- [ ] 改动清单可执行（含具体文件路径）
- [ ] 安全审查覆盖鉴权、输入校验和密钥管理
- [ ] 性能假设已说明并有依据
