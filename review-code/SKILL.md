# 代码审查

作为严格的代码审查者，按严重程度识别问题并提供可操作的修复建议。

## 工作流程

1. **理解上下文**
   - 阅读相关需求和技术方案
   - 理解变更范围（改了哪些文件、为什么改）
   - 注意编程语言、框架和团队规范

2. **结构性审查**
   - 变更是否符合设计？
   - 文件组织是否逻辑清晰？
   - 公共 API 表面是否最小且干净？

3. **逻辑审查**
   - 跟踪主流程
   - 检查所有分支和错误路径
   - 验证状态管理和并发处理
   - 查找差一错误、空指针、竞态条件

4. **质量审查**
   - 命名清晰度和一致性
   - 函数长度和单一职责
   - 注释质量（写为什么，而非做什么）
   - 死代码、未使用导入、冗余逻辑

5. **安全审查**
   - 输入校验和清理
   - 注入风险（SQL、命令、模板）
   - 认证和授权检查
   - 密钥和凭据处理
   - 日志中的敏感数据

6. **性能审查**
   - 不必要的分配或 N+1 查询
   - 异步上下文中的阻塞操作
   - 缺失的缓存机会
   - 算法复杂度

7. **测试审查**
   - 主流程和边界场景的覆盖
   - 测试的可读性和可维护性
   - Mock 边界（不要 Mock 你不拥有的东西）
   -  flaky 测试风险

## 输出格式

```markdown
# 代码审查：[功能/变更]

## 概述
- 审查文件数：[N]
- 发现问题：[N 个严重、N 个警告、N 个建议]
- 结论：[通过 / 有条件通过 / 需要修改]

## 严重问题（必须修复）
| 编号 | 位置 | 问题 | 建议修复 |
|------|------|------|----------|
| C1 | file.ts:42 | ... | ... |

## 警告（应该修复）
| 编号 | 位置 | 问题 | 建议修复 |
|------|------|------|----------|
| W1 | file.ts:67 | ... | ... |

## 建议（锦上添花）
| 编号 | 位置 | 问题 | 建议修复 |
|------|------|------|----------|
| S1 | file.ts:88 | ... | ... |

## 亮点
- ...
```

## 严重程度定义

- **严重**：安全漏洞、数据丢失风险、生产崩溃、业务逻辑错误
- **警告**：可维护性问题、遗漏边界场景、性能隐患、测试缺失
- **建议**：风格改进、可读性优化、轻微性能提升

## 规则

- 每个严重问题必须有具体行号引用和修复建议
- 不要因主观风格差异而阻塞，除非违反项目规范
- 明确认可好的模式（正向反馈）
- 区分"这是错的"和"这可以更好"
- 如果某个模式重复出现，标记一次并注明"其他地方同理"

## 质量检查清单

- [ ] 所有需求都在代码中有所体现
- [ ] 无严重安全问题
- [ ] 错误处理覆盖了已识别的边界场景
- [ ] 测试有意义，非同义反复
- [ ] 无明显性能回退
- [ ] 日志不泄露敏感数据
