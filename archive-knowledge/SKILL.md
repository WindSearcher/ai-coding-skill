# 知识归档

捕获已交付功能的完整上下文，让未来的开发者（包括未来的 AI Agent）无需重新发现决策即可理解、调试和扩展。

## 工作流程

1. **收集产物**
   - 原始 PRD / 需求分析
   - 技术方案文档
   - 最终代码变更（diff 或文件列表）
   - 代码审查反馈及处理结果

2. **记录背景**
   - 为什么需要这个功能（业务背景）
   - 解决了什么问题
   - 利益相关者是谁

3. **记录设计决策**
   - 关键架构选择及原因
   - 考虑过但拒绝的替代方案
   - 做过的权衡（性能 vs 可维护性等）
   - 对环境或用户的假设

4. **总结实现**
   - 核心模块及职责
   - 入口点和数据流
   - 外部集成及其契约
   - 配置和环境变量

5. **捕获运维知识**
   - 如何验证功能正常运行
   - 常见故障模式及诊断方法
   - 监控大盘或告警
   - 回滚流程（如适用）

6. **识别维护注意事项**
   - 已知技术债务
   - 计划中的未来改进
   - 规模增长时可能变化的部分
   - 可能需要升级的依赖

## 输出格式

```markdown
# 知识归档：[功能名称]

## 背景
- **需求来源**：[PRD 链接 / 工单]
- **业务目标**：[我们为什么做这个]
- **利益相关者**：[谁关心这个功能]

## 设计决策
| 决策 | 理由 | 已拒绝的替代方案 |
|------|------|----------------|
| ... | ... | ... |

## 实现概述
- **核心模块**：[列表及简要说明]
- **入口点**：[API、事件、CLI]
- **数据流**：[简要描述或图表链接]
- **外部依赖**：[服务、库、版本]

## 配置
| 名称 | 默认值 | 说明 |
|------|--------|------|
| ... | ... | ... |

## 运维指南
- **健康检查**：[如何验证]
- **常见问题**：[现象及修复]
- **监控**：[指标、日志、告警]

## 维护注意事项
- **技术债务**：[什么及原因]
- **未来改进**：[想法及优先级]
- **扩展考虑**：[瓶颈]
- **相关文档**：[设计、PR、运维手册链接]
```

## 存储规范

- 存放在项目的 `docs/` 或 `knowledge-base/` 目录
- 文件名带日期前缀：`YYYY-MM-功能名称.md`
- 从 README 或主索引中链接
- 代码审查时检查："知识归档是否已更新？"

## 质量检查清单

- [ ] 新开发者无需阅读全部 PRD/设计文档就能理解这个功能
- [ ] 设计决策解释了*为什么*，而非仅*做了什么*
- [ ] 运维指南包含验证和调试方法
- [ ] 维护注意事项诚实标注技术债务
- [ ] 所有外部依赖已固定版本或记录在案
- [ ] 保留了原始需求和设计的链接
