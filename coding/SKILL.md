# 编码

按照项目现有规范和提供的技术方案，实现分配的编码任务。

## 工作流程

1. **理解上下文**
   - 从任务拆解中阅读分配的任务
   - 阅读技术方案的相关章节
   - 研究目标模块及相邻文件的现有代码模式

2. **准备环境**
   - 检查是否需要安装依赖
   - 确认测试基础设施可用
   - 创建或切换到合适的分支

3. **小步增量实现**
   - 如尚未完成，先定义接口/类型
   - 实现核心逻辑
   - 添加错误处理和边界场景
   - 接入现有基础设施（路由、依赖注入、配置）

4. **同步编写测试**
   - 纯函数和业务逻辑的单元测试
   - 外部交互的集成测试
   - 确保覆盖主流程和已识别的边界场景

5. **完成前自审**
   - 运行测试套件
   - 运行代码检查工具和类型检查
   - 检查 TODO、调试日志或硬编码密钥

## 编码规范

- 精确匹配现有代码风格（格式、命名、模式）
- 优先显式而非隐式
- 快速失败并给出描述性错误
- 在合适的级别记录日志（失败用 ERROR，内部逻辑用 DEBUG）
- 注释写"为什么"，而非"做什么"
- 尽量保持函数不超过 50 行

## 输出

产出：
1. 修改/新建的源文件
2. 对应的测试文件
3. 变更简要总结：

```markdown
## 实现摘要：[任务编号]

### 变更文件
- [路径] - [变更简要说明]

### 决策说明
- [与方案偏差或意外选择的说明]

### 测试情况
- [测试运行方式及结果]

### 待办事项
- [推迟或需要注意的事项]
```

## 规则

- 除非任务间关联很琐碎，否则不要在一个会话中实现多个任务
- 如果编码过程中发现方案假设错误，记录偏差
- 绝不提交密钥、令牌或凭据
- 如果测试失败，先修复再标记完成
- 修改现有文件时，除非方案明确要求破坏性变更，否则保持向后兼容

## 质量检查清单

- [ ] 代码编译/解析无错误
- [ ] 所有新代码都有对应测试
- [ ] 测试通过
- [ ] 无新增代码检查或类型错误
- [ ] 需求中的边界场景已处理
- [ ] 无本应是配置项的硬编码值
- [ ] 日志恰当且不嘈杂
