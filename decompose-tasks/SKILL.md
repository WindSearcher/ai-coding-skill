# 任务拆解

将技术方案拆分为最小可独立实现的编码单元，支持并行开发并明确依赖关系。

## 工作流程

1. **分析方案**
   - 阅读技术方案文档
   - 识别所有需要新建或修改的文件
   - 注意跨模块依赖

2. **确定任务粒度**
   - 每个文件新建或重大修改对应一个任务
   - 将琐碎变更归组（如配置更新、导入修复）
   - 当多个模块依赖某接口时，先独立定义接口再实现

3. **定义任务元数据**
   - **文件路径**：相对于项目根目录的精确路径
   - **改动类型**：`新建`、`修改`、`删除`、`重构`
   - **复杂度**：`低`（<2 小时）、`中`（2-4 小时）、`高`（>4 小时或高风险）
   - **依赖**：必须先完成的任务的编号
   - **说明**：实现内容描述和关键注意事项

4. **排列任务顺序**
   - 第 1 层：基础设施、接口定义、Schema 变更（基础层）
   - 第 2 层：核心业务逻辑（依赖第 1 层）
   - 第 3 层：展示层、API、消费者（依赖第 2 层）
   - 第 4 层：测试、文档、打磨

5. **识别并行机会**
   - 同层且无相互依赖的任务可并行执行
   - 标记适合结对编程或需高级审查的任务

## 输出格式

```markdown
# 任务拆解：[功能名称]

## 概述
- 总任务数：[N]
- 预估总工时：[X 小时]
- 最大并行深度：[N 层]

## 第 1 层：基础
| 编号 | 文件 | 类型 | 复杂度 | 依赖 | 说明 |
|------|------|------|--------|------|------|
| T-01 | ... | 新建 | 低 | 无 | ... |

## 第 2 层：核心逻辑
| 编号 | 文件 | 类型 | 复杂度 | 依赖 | 说明 |
|------|------|------|--------|------|------|
| T-02 | ... | 新建 | 中 | T-01 | ... |

## 第 3 层：接口与集成
...

## 第 4 层：测试与打磨
...

## 依赖图
```
T-01 -> T-02 -> T-04
     -> T-03 -> T-05
```

## 风险标记
| 任务编号 | 风险 | 缓解措施 |
|----------|------|----------|
| ... | ... | ... |
```

## 规则

- 有依赖的任务必须等所有依赖都合并/审查后才能开始
- `新建`文件的接口应先定义好，再启动依赖它的`修改`任务
- `高`复杂度任务应在说明中包含微型设计
- 如果一个任务要修改超过 3 个文件，除非变更很琐碎，否则应拆分

## 质量检查清单

- [ ] 技术方案中的每个文件都有对应任务
- [ ] 任务间没有循环依赖
- [ ] 每个任务有单一、明确的交付物
- [ ] 复杂度分布合理（不全是高）
- [ ] 所有核心逻辑变更都有对应的测试任务

## 上下文传递

### 输入
- 技术方案产物: `docs/design/{功能名称}.tech-design.md`
- 需求分析产物（可选，用于理解业务背景）: `docs/requirements/{功能名称}.requirements.md`

### 输出
- 保存为: `docs/tasks/{功能名称}.tasks.md`
- 下一阶段的输入来源: 任务表（按层级），每个任务的编号、文件路径、依赖关系
